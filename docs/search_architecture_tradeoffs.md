# 검색 아키텍처 트레이드오프: BM25 vs 벡터 검색 vs LLM 직접 검색

**작성일**: 2026-03-26

---

## 배경

볼트 1,490개 문서를 대상으로 BM25 검색 수정(B=0.75→0.3, 파일명 boost) 후,
"그냥 MD 파일 전체를 LLM에 넘겨 검색시키는 것과 차이가 있나?"라는 의문에서 출발.

---

## "전체 MD → LLM 검색"이 불가능한 이유

볼트 규모: 1,490개 문서 × 평균 2,000자 = **약 300만 자 = 75만 토큰**

- Claude 최대 컨텍스트(200K)에도 들어가지 않음
- 들어간다 해도 쿼리 1회에 수천 원
- 실시간 검색 불가

---

## BM25가 실제로 하는 일

```
1,490개 문서 → BM25 필터 → 상위 10개 → LLM이 읽음
```

LLM은 BM25가 선별한 상위 10개(~20,000자)만 처리한다.
**BM25는 LLM의 눈 역할** — LLM 직접 검색과 BM25+LLM은 목적이 다름.

---

## 진짜 비교 대상: BM25 vs 벡터 임베딩 검색

| 항목 | BM25 (현재) | 벡터 임베딩 |
|------|-------------|-------------|
| 매칭 방식 | 키워드 토큰 매칭 | 의미 유사도 |
| "월영 최신 현황" 쿼리 | "월영"+"현황" 토큰 매칭 | 의미적으로 관련 문서 탐색 |
| 속도 | 수 ms | 수십 ms |
| 비용 | 무료 | 임베딩 API 비용 발생 |
| 문서 추가 시 | 즉시 재인덱싱 | 임베딩 재생성 필요 |
| 구현 복잡도 | 낮음 | 벡터 DB + 임베딩 모델 필요 |
| 길이 긴 문서 | 페널티 (BM25_B로 조정) | 길이 무관 |

---

## BM25의 구조적 한계가 드러난 실제 케이스

**쿼리**: "월영 최신 작업 상태"
**기대 문서**: `652885523_월영(Wolyoung).md` (SFX+보이스 전체 스펙)
**현상**: BM25 top 5에 미진입 → RAG 컨텍스트에서 완전 누락

원인: 대형 테이블 문서의 길이 정규화 페널티.
벡터 검색이었다면 "최신 작업 상태 ≈ 사운드 스펙 문서"를 의미론적으로 연결했을 것.

→ BM25_B 0.3 + 파일명 boost로 임시 해결했으나, 추상적 쿼리의 구조적 한계는 남아 있음.

---

## 현재 판단

1,490개 규모에서 BM25는 충분히 실용적이며, 이번 수정으로 대부분의 케이스가 해결됨(100개 시나리오 97% 정확도).

벡터 검색 전환을 고려할 시점:
- 문서 수가 5,000개 이상으로 증가할 때
- "추상적 의미 쿼리"(현황, 진행 상태, 방향성 등)의 실패율이 높아질 때
- 다국어 혼용 쿼리가 많아질 때

지금 당장 전환은 오버엔지니어링.
