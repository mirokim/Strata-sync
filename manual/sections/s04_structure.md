> **5. 대용량 문서 분할 --- 허브-스포크 구조**

단일 대형 문서를 그대로 투입하면 BFS 탐색 시 한 노드에 토큰이 집중되어
비효율적이다. 허브 문서 + 스포크 문서로 분할한다.

**5.1 분할 기준**

  ---------------------------------- --------------------------------------------------
  **기준**                           **설명**
  섹션이 독립적으로 의미가 있을 때   개별 탐색이 필요한 하위 주제로 분리
  단일 섹션이 200줄 이상일 때        단독 파일로 분리하여 BFS 토큰 효율화
  섹션 간 교차 참조가 많을 때        위키링크로 연결하면 그래프 엣지가 생겨 탐색 개선
  ---------------------------------- --------------------------------------------------

**5.2 허브 문서 패턴**

허브 문서는 목차 역할만 하며, 실제 내용은 스포크 문서에 위임한다.

> \-\--
>
> tags: \[주제명, 허브\]
>
> date: YYYY-MM-DD
>
> \-\--
>
> \# 주제명
>
> \> 한줄 요약
>
> \#\# 목차
>
> \- \[\[주제명 - 섹션A\|1. 섹션A\]\]
>
> \- \[\[주제명 - 섹션B\|2. 섹션B\]\]

**5.3 허브 파일 크기 관리 기준**

허브 파일이 과도하게 커지면 PageRank가 한 노드에 집중되어 Graph RAG 탐색이
왜곡된다. 아래 기준 중 하나라도 충족되면 분할을 검토한다.

  ----------------------------------------- -------------------------------------------------------
  **지표**                                  **분할 검토 임계값**
  인바운드 링크 수                          50개 초과
  파일 본문 줄 수                           300줄 초과
  단일 허브에서 나가는 outbound 링크 수     30개 초과
  ----------------------------------------- -------------------------------------------------------

분할 절차:

1. 분할 대상 허브를 주제별 서브허브(예: `허브명_A.md`, `허브명_B.md`)로 분리

2. 원본 허브는 목차 역할만 남기고 서브허브로 링크 위임 (§5.2 패턴 적용)

3. 기존에 원본 허브를 가리키던 링크 중 세부 주제에 해당하는 것은 서브허브로 교체

4. check\_quality.py 재실행 → 고립 노드 0 확인

> *chief persona 파일이 인바운드 링크 254개 → 9개로 줄어든 사례(v3.4)처럼,
> 허브 분할 후 PageRank 분산 효과를 확인하는 것을 권장한다.*

> **6. 프론트매터 통일**

**6.1 필드 정의**

> \-\--
>
> date: YYYY-MM-DD \# 필수 --- 최신도 판단 기준 (recency boost 적용: 파일명 날짜 > date > mtime, §18.6)
>
> type: spec \# spec / decision / meeting / guide / reference
>
> status: active \# active / outdated / deprecated
>
> tags: \[도메인, 세부주제\] \# 핵심 도메인 1 + 세부 주제 1\~2
>
> source: \"https://\...\" \# 필수 --- Confluence/Jira/Notion 등 원본
> URL. 자동 생성 파일은 \"generated\" 기재
>
> origin: confluence \# 출처 플랫폼 식별자
> \# html / pdf / pptx / xlsx / docx / txt / md / generated
>
> \-\--

**6.1.1 origin: generated 파일 처리 규칙**

파이프라인이 자동 생성하는 파일들은 외부 URL이 없으므로 `source: generated`을
명시하고, check\_quality.py에서 source URL 누락으로 오탐하지 않도록 예외 처리한다.

  ----------------------------- ------------------------------------------
  **자동 생성 파일**            **origin: generated 적용 대상**
  chief persona.md              최상위 허브. gen\_year\_hubs.py 생성
  \_index.md                    전체 인덱스. gen\_index.py 생성
  currentSituation.md           현재 상황 컨텍스트. 수동 작성·갱신
  회의록\_YYYY.md               연도별 허브. gen\_year\_hubs.py 생성
  태그\_정의서.md               태그 정의 목록. 수동 작성·갱신
  ----------------------------- ------------------------------------------

> *check\_quality.py의 source URL 누락 점검 시 `origin: generated` 파일은
> 제외하도록 스크립트를 구현할 것.*

**6.2 type 분류 기준**

  ------------- --------------------------------------------------
  **type 값**   **적용 문서 유형**
  spec          기능 설계서, 기술 스펙, 요구사항 문서
  decision      의사결정 기록, ADR(Architecture Decision Record)
  meeting       회의록, 이사장 피드백, 정례보고 기록
  guide         사용 가이드, 온보딩 문서, 매뉴얼
  reference     외부 참고 자료, 벤치마크, 연구 자료
  ------------- --------------------------------------------------

**6.3 태그 지정 절차**

-   팀 공용 태그 목록(태그\_정의서.md) 먼저 확인

-   없으면 자유 입력 후 태그\_정의서.md에 추가

-   핵심 도메인 1개 + 세부 주제 1\~2개 조합이 기본

> *특수 역할 태그(예: chief)는 Graph RAG 쿼리 필터 기준이 된다. 해당
> 인물·주제 관련 모든 문서에 일관되게 태그를 달아야 피드백 아카이브
> 탐색이 정상 동작한다.*

**6.3.1 신규 태그 추가 및 태그\_정의서 갱신 절차**

기존 태그로 커버되지 않는 개념이 등장할 때 아래 절차로 신규 태그를 등록한다.

1. **필요성 판단**: 동일 의미의 태그가 태그\_정의서.md에 없는지 먼저 확인.
   유사 태그가 있으면 신규 추가 대신 기존 태그를 확장하는 것을 우선 검토.

2. **네이밍 규칙**: 소문자 영문 또는 한글, 하이픈 구분 (`game-design`),
   2단어 초과 금지. 단수형 원칙.

3. **태그\_정의서.md 갱신**: 신규 태그를 아래 형식으로 추가.

   > `태그명` \| 적용 대상 설명 \| 예시 파일명

4. **기존 문서 소급 적용**: 신규 태그가 적용되어야 할 기존 문서를 검색하여
   frontmatter tags 필드에 추가. (`grep -rl "키워드" active/` 활용)

5. **chief 등 특수 태그**: §11 기준에 따라 별도 관리. 태그\_정의서.md에
   특수 태그임을 명시하고 남용 방지 기준을 기재.

> *태그 수가 과도하게 늘어나면 Graph RAG 필터 정확도가 떨어진다.
> 분기별 태그\_정의서.md 검토 시 유사·중복 태그를 통합한다 (§17.2 참조).*
