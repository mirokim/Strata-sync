**18.5 Confluence 배치 동기화 date 오염 대응**

Confluence 스페이스를 일괄 내보내거나 페이지를 대량 이동하면, Confluence의
`version.when` 필드가 **실제 작성일이 아닌 동기화 실행 날짜로 일괄 덮어씌워진다.**
refine\_html\_to\_md.py는 이 값을 그대로 `date:` 필드로 사용하므로 2021년 문서도
`date: 2026-03-11` 처럼 최근 날짜로 찍힐 수 있다.

**18.5.1 오염 판별**

배치 날짜 오염의 특징: 특정 날짜를 가진 파일이 **5개 이상 집중**된다.

> python check\_outdated.py \<active\_dir\> \--batch-check

또는 수동 확인:

> grep -rh "^date:" active/ | sort | uniq -c | sort -rn | head -20
>
> → 동일 날짜가 10개 이상이면 배치 동기화 날짜로 의심

실제 오염 사례 (refined\_vault 2026-03-13 진단):

  -------------------- ------- -----------------------------------------------------------------
  **오염 날짜**        **수**  **원인**
  `2026-03-11`         329개   최근 Confluence 배치 동기화. 2020\~2025년 문서 포함.
  `2024-02-28`         69개    이전 Confluence 배치 동기화. 2021년 문서 포함.
  `2022-05-17`         14개    그 이전 배치 동기화.
  -------------------- ------- -----------------------------------------------------------------

**파급 효과**: gen\_year\_hubs.py가 오염된 date 기준으로 연도 허브를 생성하면,
`회의록_2026.md` "최근 추가 5개" 섹션에 **2021년 문서**가 올라간다.
BFS가 이 허브를 출발점으로 삼으면 첫 Hop부터 2021년 문서로 탐색 → 구버전 응답.

**18.5.2 수정 절차**

파일명에서 날짜를 추출하여 오염된 `date:` 필드를 덮어쓴다.

**(1) 파일명 날짜 추출 패턴 (우선순위 순)**

  --------------------------------- ------------------------------------- -------------------------
  **파일명 패턴**                   **예시**                              **추출 날짜**
  `[YYYY_MM_DD]` 대괄호 형식        `[2022_02_07] 이사장 피드백.md`       2022-02-07
  `YYYYMMDD` 8자리                  `ProjectA_정례_20240215_월영.md`      2024-02-15
  `_YYMMDD_` 또는 `_YYMMDD.` 6자리  `ProjectA_월영연출_250714.md`         2025-07-14
  파일명 내 연도만 (`_YYYY_`)       `정례보고 자료_2022.md`               2022-01-01 (연도만)
  --------------------------------- ------------------------------------- -------------------------

*6자리 패턴(YYMMDD)은 20YY로 해석하되, 2018\~2030 범위에서만 유효로 처리한다.*

**(2) fix\_dates.py 스크립트 실행**

> \# dry-run: 수정 예상 목록만 출력
>
> python fix\_dates.py \<active\_dir\>
>
> \# 실제 수정
>
> python fix\_dates.py \<active\_dir\> \--apply

  --------------------------------- -------------------------------------------------------------------
  **옵션**                          **설명**
  `--batch-dates 2026-03-11,...`    오염으로 간주할 날짜 목록 (쉼표 구분). 기본값: 자동 감지 (5개+)
  `--apply`                         실제 파일 수정. 없으면 dry-run.
  `--no-match-log out.txt`          파일명에서 날짜 추출 실패한 파일 목록 저장 (수동 처리용)
  --------------------------------- -------------------------------------------------------------------

**(3) 추출 실패 파일 처리**

파일명에 날짜가 전혀 없는 경우(Confluence ID만 있는 파일 등):

1.  title 필드에 `[YYYY.MM.DD]` 패턴이 있으면 그 값을 사용
2.  없으면 파일 본문 첫 날짜 언급을 수동으로 확인
3.  그것도 없으면 **연도만 추출 가능한 경우** `YYYY-01-01`로 대략 입력
4.  완전 불명인 경우 `date: unknown` 표시 후 수동 처리 대기

**18.5.3 수정 후 재생성 순서**

date 필드 일괄 수정 후 **반드시** 아래 순서로 재생성한다.

> \# 1. 키워드 주입 (날짜 변경으로 인한 recency boost 재계산 불필요, 링크 확인용)
>
> python inject\_keywords.py \<active\_dir\>
>
> \# 2. 전체 인덱스 재생성
>
> python gen\_index.py \<active\_dir\>
>
> \# 3. 연도별 허브 재생성 (오염된 날짜 기반으로 만들어진 허브 교체)
>
> python gen\_year\_hubs.py \<active\_dir\> \--top 5

*gen\_year\_hubs.py가 올바른 날짜 기준으로 연도 허브를 재생성해야 `회의록_2026.md`
"최근 추가 5개" 섹션에 **실제 최신 문서**가 올라간다.*

**18.5.4 재발 방지**

-   Confluence에서 **페이지 이동·스페이스 재편** 작업 후에는 반드시 check\_outdated.py
    \--batch-check 실행
-   대량 동기화 직후에는 `grep ... | uniq -c | sort -rn` 으로 date 분포를 확인하고
    동일 날짜 10개 이상이면 fix\_dates.py 우선 실행
-   Recency boost(§18)는 `date:` 필드(frontmatter)를 기준으로 작동하며,
    파일 시스템 mtime은 사용하지 않는다. Confluence 동기화 시 mtime이 오늘로
    갱신되어도 `date:`만 올바르면 문제 없다.
-   **HTML 변환 스크립트(refine\_html\_to\_md.py) 날짜 우선순위 설정 확인 (근본 재발 방지)**:
    스크립트가 Confluence HTML 메타에서 날짜를 읽을 때 `created_date`(생성일)를
    `modified_date`(수정일)보다 **반드시 우선 사용**해야 한다.
    수정일은 페이지 이동·재구성 시 오늘 날짜로 갱신되지만, 생성일은 변하지 않기
    때문이다.

    확인 방법: `refine_html_to_md.py` 내 date 결정 로직이
    `date = created_date or modified_date or today` 순서인지 점검.
    `modified_date or created_date` 순서면 반드시 수정한다.

**18.6 검색 파이프라인 mtime 문제 및 Recency Boost (v3.26)**

Confluence 볼트를 일괄 동기화하면 **모든 파일의 파일시스템 mtime이 동기화 시점으로 통일**된다.
이 때문에 mtime 기반 recency boost는 문서 간 차별화가 불가능하다.

실제 사례 (refined\_vault 2026-03-23 진단):

  ------------------- -------
  **mtime**           **수**
  2026-03-23 15:44    전체 1,210개 (active 전부)
  ------------------- -------

> 이사장 피드백 40여 개 파일이 전부 동일 mtime → recency boost 적용해도 순위 변화 없음.

**18.6.1 해결: 파일명 날짜 파싱 (`parseFilenameDate`)**

graphAnalysis.ts에 `parseFilenameDate()` 함수를 추가하여 파일명에서 콘텐츠 작성일을 추출한다.
mtime 대신 이 값을 recency boost에 사용한다.

파싱 우선순위: **파일명 날짜 > frontmatter date > 파일시스템 mtime**

  --------------------------------- ------------------------------------- -------------------------
  **파일명 패턴**                   **예시**                              **추출 날짜**
  `YYYY_MM_DD` / `YYYY-MM-DD`      `_2025_08_18.md`, `[2022_02_07]`      2025-08-18 / 2022-02-07
  `YYYYMMDD` 8자리 (2000\~2039)    `ProjectA_정례_20240215_월영.md`      2024-02-15
  `YY_MM_DD` 밑줄/하이픈 구분       `25_03_28 시연용.md`                  2025-03-28
  `YYMMDD` 6자리                   `ProjectA_월영연출_250714.md`         2025-07-14
  --------------------------------- ------------------------------------- -------------------------

**주의**: Notion/Confluence ID가 8자리 숫자인 경우 오파싱 방지를 위해 YYYYMMDD 패턴은
연도 2000\~2039로 제한한다. 미래 날짜는 30일 이내만 허용한다.

**18.6.2 Recency Boost 수식**

> `score *= 1 + 0.1 * exp(-daysOld / 180)`

- 6개월 이내: \~10% 부스트
- 1년 경과: \~3%
- 2년 이상: 거의 0 (기존 점수 유지)

directVaultSearch와 BM25 search 양쪽에 동시 적용된다.

**18.6.3 검증 결과 (200개 시나리오)**

- "이사장님 최근 피드백" → 2026-01-07 문서 5위→**1위**
- "2026년 1월 작업 내용" → 26-01-19 주간업무보고 상위 진입
- 200개 시나리오 97% 통과 (194/200)

**18.6.4 정제 시 주의사항**

- 파일명에 날짜가 없는 문서(Confluence ID만 있는 파일)는 recency boost 미적용.
  fix\_dates.py로 frontmatter date를 올바르게 설정하면 2순위로 적용됨.
- frontmatter date 오염(§18.5)이 있으면 recency boost도 오염됨.
  **fix\_dates.py를 먼저 실행**한 뒤 검색 테스트.
- TFIDF\_SCHEMA\_VERSION이 변경됨 (6→7). 볼트 재오픈 시 BM25 캐시가 자동 무효화되어 재빌드.

**18.6.5 BM25 범용어 과잉 매칭 잔존 이슈**

coverage^0.5 보정으로 대부분 해소되었으나, 아래 패턴은 여전히 발생:

  --------------------------------- -------------------------------------------------------------------
  **쿼리**                          **문제**
  "마투아 지역 설정"                "지역", "설정" 범용어 → 정례보고가 상위 매칭
  "퀘스트 시나리오"                  둘 다 범용어 → "기획\_" 문서에 분산 매칭
  --------------------------------- -------------------------------------------------------------------

향후 PPR rerank 적용 시 해소 예정. 현재 검색에서는 **고유명사 + 범용어** 조합이 가장 효과적.

**17.4 롤백 전략 (git 없는 환경)**

inject\_keywords.py 등 대량 수정 스크립트 실행 후 문제가 발생했을 때의 복원 절차.

**사전 준비 (스크립트 실행 전 필수):**

```bash
# 실행 전 백업 생성 (zip)
python -c "
import shutil, datetime
ts = datetime.datetime.now().strftime('%Y%m%d_%H%M%S')
shutil.make_archive(f'vault_backup_{ts}', 'zip', 'active/')
print(f'백업 생성: vault_backup_{ts}.zip')
"
```

또는 스크립트 내 자동 백업 옵션 사용:
```bash
python inject_keywords.py <active_dir> --backup  # 실행 전 자동 backup/ 폴더 생성
```

**복원 방법:**

| 상황 | 복원 절차 |
|------|-----------|
| 스크립트 실행 직후 이상 발견 | `backup/` 폴더 → `active/` 교체 (스크립트가 --backup 옵션 지원 시) |
| ZIP 백업에서 복원 | `unzip vault_backup_YYYYMMDD_HHMMSS.zip -d active_restored/` |
| 특정 파일만 복원 | `python restore_file.py <stem>` — backup/ 에서 해당 파일만 복사 |
| 중첩 wikilink 발생 | `audit_and_fix.py --fix` (중첩 링크 자동 수정 ①번 항목) |

**스크립트별 복원 명령:**

```bash
# inject_keywords.py 오주입 일괄 복원
# [[target_stem|키워드]] → 키워드 (plain text)
python inject_keywords.py <active_dir> --revert --keyword "퀘스트"

# 특정 날짜 이후 변경된 파일 목록 확인
find active/ -newer vault_backup_20260319_120000.zip -name "*.md"
```

> *git이 없는 환경에서도 스크립트 실행 전 ZIP 백업을 생성하는 것을 팀 표준으로 정한다.
> 백업 파일 보관 기간: 최소 72시간 (문제 발견 후 복원 여유 확보).*

**충돌 유형 및 해결 기준:**

| 충돌 유형 | 발생 조건 | 해결 방법 |
|-----------|-----------|-----------|
| Frontmatter 필드 충돌 | 두 사람이 같은 파일의 tags·status 수정 | 나중 수정 기준 (최신 date 우선). status는 더 낮은 것 우선 (active > outdated) |

> **19. AI 보조 정제 시 컨텍스트 한계 대응**

Claude Cowork, Claude Code 등 AI 도구로 정제 작업을 수행할 경우, **컨텍스트 윈도우 제한** 때문에 다단계 파이프라인이 중간에 끊어지는 문제가 발생한다. 아래 절차를 따르면 세션이 초기화되어도 작업을 안전하게 이어갈 수 있다.

**19.1 문제 현상**

- 대부분의 AI 도구(Claude 포함)는 컨텍스트 용량 초과 시 새 세션을 시작하며, 이전 세션의 작업 상태를 기억하지 못한다.
- 14단계 파이프라인을 한 세션에서 전부 완료하는 것은 현실적으로 불가능하다.
- AI가 자동으로 "다음 단계"로 넘어가지 못하고 멈추는 경우가 발생한다.

**19.2 세션 재시작 시 표준 복구 절차**

새 세션이 시작되면 **반드시 아래 순서로 작업 상태를 파악**한 뒤 작업을 이어간다.

  ------ -------------------------------------------------------------------
  **순서** **작업**
  1      check\_quality.py 실행 → 현재 품질 지표 확인 (어디까지 진행됐는지 파악)
  2      본 매뉴얼 전체 검토 → 구현된 스크립트와 미구현 항목 대조
  3      .tools/ 폴더 목록 확인 → 생성된 스크립트 목록으로 진행 상황 재확인
  4      미구현 항목 중 우선순위 높은 것부터 재개
  ------ -------------------------------------------------------------------

**19.3 세션 이어받기용 프롬프트 템플릿**

새 세션 시작 시 아래 프롬프트를 AI에게 전달하면 빠르게 상태를 파악하고 작업을 이어간다.

> "Graph RAG 데이터 정제 매뉴얼 v3.26을 읽고 현재 구현 상태를 파악한 다음, 미구현 항목을 우선순위 순서대로 진행해줘.
> check\_quality.py로 현재 품질 지표를 먼저 확인하고(헤딩 비율 99% 기준), .tools/ 폴더의 스크립트 목록도 확인해줘.
> v3.26 변경사항: 검색 파이프라인에 recency boost 적용됨. parseFilenameDate()로 파일명에서 날짜 추출 → getContentDate() 우선순위(파일명 > frontmatter date > mtime). BM25/directVaultSearch 모두 score \*= 1 + 0.1 \* exp(-daysOld/180) 적용. TFIDF\_SCHEMA\_VERSION 7. Confluence 동기화 후 fix\_dates.py 실행 권장(§18.5·§18.6 참조)."

**19.4 중간 체크포인트 기록 (권장)**

장기 정제 작업 중에는 아래 내용을 메모해두면 세션 재시작 후 빠르게 복구된다.

**19.5 자동화 우선 원칙**

수동 수정이 필요한 경우라도 **반드시 스크립트화하여 .tools/에 저장**해두어야 한다. 세션이 끊어졌을 때 수동 작업은 재현이 불가능하지만, 스크립트는 언제든 재실행할 수 있다.

**20.3 MD → DOCX 동기화 절차**

**방법 A: pandoc 자동 변환 (권장)**

MD → DOCX를 가장 안전하게 변환하는 방법이다. XML 수동 편집 오류를 방지한다.

```bash
# 기본 변환 (스타일 참조 없음)
pandoc "Graph RAG 데이터 정제 매뉴얼 v3.23.md" \
  -o "Graph RAG 데이터 정제 매뉴얼 v3.23.docx"

# 기존 DOCX 스타일 템플릿 적용 (권장 — 기존 포맷 유지)
pandoc "Graph RAG 데이터 정제 매뉴얼 v3.23.md" \
  --reference-doc="Graph RAG 데이터 정제 매뉴얼 v3.22.docx" \
  -o "Graph RAG 데이터 정제 매뉴얼 v3.23.docx"
```

pandoc 설치:
- Windows: `winget install JohnMacFarlane.Pandoc` 또는 [공식 다운로드](https://pandoc.org/installing.html)
- macOS: `brew install pandoc`
- Ubuntu: `sudo apt install pandoc`

변환 후 검증:
```bash
# 주요 신규 문구가 포함됐는지 확인
pandoc "Graph RAG 데이터 정제 매뉴얼 v3.23.docx" -t plain | grep "Tesseract"
pandoc "Graph RAG 데이터 정제 매뉴얼 v3.23.docx" -t plain | grep "멀티 유저"
```

**방법 B: XML 수동 편집 (pandoc 미사용 환경)**

Claude Cowork(또는 Claude Code)에서 DOCX 스킬을 사용하여 아래 절차로 진행한다.

  ------ -----------------------------------------------------------------------
  **순서** **작업**
  ①      `scripts/office/unpack.py` 로 기존 DOCX 언패킹 → `/manual_unpacked/`
  ②      `word/document.xml` 에서 변경된 섹션을 찾아 XML 편집
  ③      `scripts/office/pack.py` 로 리패킹 → `vX.Y.docx` 저장
  ④      pandoc 추출 후 키워드 grep 검증 (주요 신규 문구가 포함됐는지 확인)
  ------ -----------------------------------------------------------------------

> 편집 시 `<w:t>` 태그 안의 텍스트만 수정하며, `<w:pStyle>` 값은 기존 단락
> 스타일을 유지한다. 신규 단락 삽입 시 인접 단락 블록을 복사하여 스타일을 상속한다.

> *방법 A(pandoc)를 우선 시도하고, 스타일 불일치가 심각한 경우에만 방법 B를 사용한다.
> XML 수동 편집은 태그 구조 오류 시 DOCX가 열리지 않을 수 있으므로 수정 전 DOCX 원본을 백업한다.*
